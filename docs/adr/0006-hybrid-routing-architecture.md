# 0006 — 混合动态路由架构

Agent 协作采用三层混合架构：调度器宏观编排 + Agent 微观自治 + 条件跳转效率优化。

## 背景

当前 AutoRunner 是固定管线：Writer → Critic → Manager，唯一的变化是重试循环和 Director 的后置分析。这导致：
- 高潮章节和过渡章节走同样的流程，浪费 Token
- Writer 发现设定缺失时无法主动查询，只能靠 ContextAssembler 的静态注入
- Critic 发现局部硬伤时必须走全量重试，无法触发局部修复

## 决策

1. **调度器（宏观编排）**：基于章节类型（高潮/过渡/日常）和健康度诊断，动态决定 Agent 出场顺序。例如高潮章节强制前置 Director 分析张力，日常章节直接跳过 Director。实现为 AutoRunner 内的路由逻辑。
2. **Agent 自治（微观自决）**：赋予 Agent 工具调用权。Writer 可主动挂起并调用 Retriever 查询缺失设定；Critic 可触发 Writer 的"局部热修复"，而非走 AutoRunner 的全量重试。
3. **条件跳转（效率优化）**：管线中的 `if/else` 短路逻辑。例如 Critic 评分 > 90 时跳过 Manager 即时状态提取，改为批处理。
4. **安全围栏**：Agent 自治必须在调度器设定的约束内运行，防止无限递归调用。约束包括：单次自治调用的 Token 预算上限、核心 Canon 不可违背、最大递归深度、超时熔断。

## 考虑过的替代方案

- **纯固定管线**：现状。简单但低效，无法适应不同章节的需求。
- **Agent 完全自治**：风险高。Agent A 调 Agent B，Agent B 又调 Agent A，容易形成死循环。
- **纯调度器控制**：调度器需要对所有场景做决策，复杂度爆炸。

## 影响

- AutoRunner 需要重构为可分支的管线
- Agent 基类需要支持工具调用接口
- 需要定义安全围栏的约束参数和熔断机制
- 调度器的决策逻辑需要可配置（通过 novel.yaml 或 Director 指导）
