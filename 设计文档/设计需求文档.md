# 📖 L.O.O.M. V1.0.1 产品需求文档 (PRD)

## 一、 产品愿景与定位
L.O.O.M. 是一个**本地优先的叙事操作系统**，旨在为长篇小说创作者提供抗遗忘、高一致、低摩擦的 AI 辅助创作环境。
它不是“一键生成小说”的玩具，而是作者的“数字织机”与“记忆外脑”。核心目标是：**让作者只专注于用自然语言创作，由系统在底层维护世界观的严丝合缝。**

## 二、 核心设计原则
1. **Human-first (人类主导)**：作者只写纯文本 Markdown，严禁向作者暴露或要求填写 YAML 配置。
2. **Memory-as-shadow (记忆如影)**：结构化数据（YAML/SQLite）是 AI 自动提取的影子，随正文而生，不作为输入层。
3. **Zero-Trust AI (零信任输出)**：AI 提取的状态变更必须经过人工 Diff 审阅才可固化，支持一键回滚。
4. **Model Agnostic (模型无关)**：不绑定特定大模型，通过总线屏蔽底层差异。

## 三、 核心功能需求

### 3.1 项目初始化 (`loom init`)
- **描述**：在当前目录生成标准小说项目结构。
- **输出**：`canon/`, `characters/`, `draft/`, `outlines/`, `subconscious/`, `.snapshots/`, `loom.yaml`。
- **规则**：生成的角色和设定模板必须包含 `id` 字段，强制建立 Canonical IDs 规范（如 `char_001`）。

### 3.2 沉浸式续写 (`loom write`)
- **描述**：启动交互式写作循环，Actor 代理根据上下文续写剧情。
- **触发**：用户输入空行或特定指令。
- **上下文组装规则**：
  - 强制注入权威分级标签：`[CANON | IMMUTABLE]` > `[STATE MEMORY | MEDIUM]` > `[SUBCONSCIOUS FRAGMENT | LOW]`。
  - 启用 Token 熔断机制：超过预算时，优先裁剪低权威的潜意识碎片。
- **输出**：流式输出纯文本，追加到当前 Markdown 正文区。

### 3.3 灵感潜意识池 (`loom stash`)
- **描述**：随时收录零散灵感（台词、画面、象征物）。
- **输入**：`loom stash "深渊不收我" #金句 #绝望`
- **行为**：将文本追加至 `subconscious/lines.md`，并触发后台增量向量化索引。

### 3.4 状态审阅与固化 (`loom commit`)
- **描述**：章节写完后，提取时间线、角色状态、因果事件，并要求人工确认。
- **前置动作**：自动生成当前状态的 `.snapshot.json`。
- **AI 提取**：Auditor 代理输出结构化 JSON（包含事件类型、涉及 ID、因果压强）。
- **人工审阅**：在终端展示高亮 Diff（如 `+ char_001 左臂骨折`），要求用户输入 `[y/n/edit]`。
- **固化**：确认后，更新对应角色的 YAML Frontmatter，并将事件写入 SQLite 账本。

### 3.5 状态回滚 (`loom rollback`)
- **描述**：撤销上一次或指定的 commit 操作，恢复错误提取的状态。
- **行为**：读取 `.snapshots/` 中的前置状态，覆盖当前 YAML 和 SQLite 记录。

### 3.6 一致性校验 (`loom diff`)
- **描述**：比对当前正文内容与机器影子状态的一致性。
- **行为**：检测“正文写伤愈，但 YAML 仍记为受伤”等逻辑漏洞，输出 Mismatch 警告。

### 3.7 世界观诊断 (`loom doctor`) [V1 钩子]
- **描述**：诊断世界线健康度。
- **行为**：V1 版本仅实现基础检测（孤立角色、时间线倒错），为 V2 的 Weaver 代理留出接口。

# 🏗️ L.O.O.M. V1.0.1 技术架构设计文档

## 一、 架构概览：三层解耦
L.O.O.M. 采用严格的三层解耦架构，确保创作自由度与机器严格性的平衡：
1. **Human Layer (创作层)**：纯 Markdown 文件。保证人类永远拥有 Git 版控、Obsidian 阅读、VS Code 编辑的绝对自由。
2. **Machine Shadow (状态层)**：YAML Frontmatter (局部即时状态缓存) + SQLite (全局事件账本) + Snapshots (时间机器)。对人类不可见/无需关心。
3. **Semantic Layer (语义层)**：LlamaIndex + BGE-M3 向量索引。将历史与设定转化为可召回的活体记忆。

## 二、 技术栈定版
| 分类 | 依赖库 | 职责 |
| :--- | :--- | :--- |
| **CLI 壳** | `Typer`, `Rich` | 命令路由、终端富文本/Diff 渲染 |
| **LLM 总线** | `LiteLLM`, `tenacity` | 统一模型调用、限流与重试容错 |
| **检索引擎** | `LlamaIndex` | 文档解析、向量索引、语义路由 |
| **向量化** | `BGE-M3` (本地) | 高质量中英文本向量化，防数据泄漏 |
| **数据校验** | `Pydantic V2`, `SQLModel` | 状态结构强类型校验，ORM 映射 |
| **状态账本** | `SQLite` | 全局因果事件日志存储 |
| **文件解析** | `python-frontmatter` | Markdown 与 YAML 的安全读写隔离 |
| **工具集** | `tiktoken`, `orjson`, `networkx` | Token 熔断计算、高速序列化、因果图预留 |

## 三、 核心数据模型

### 3.1 Canonical ID 规范
系统内一切关联（YAML属性、SQL外键）**严禁使用角色名**，必须使用稳定 ID。
- 角色：`char_001`
- 地点：`loc_london`
- 物品：`item_clock`

### 3.2 Markdown + YAML Shadow 结构
以角色档案为例，作者只编辑正文区，Auditor 只编辑 Frontmatter 区。
```markdown
---
id: char_001
name: Alice
aliases: [Alys]
current_state:
  location: loc_tower
  physical: [left_arm_fracture] # 机器可读的状态码
  emotional: { grief: 0.7, anger: 0.2 }
---

# Alice 的背景故事
(作者自由书写的区域...)
```

### 3.3 SQLite Event Ledger (事件账本)
用于跨章节溯源和 Phase 2 因果推演。
```sql
class EventLog(SQLModel, table=True):
    event_id: str # 唯一标识
    chapter_id: str
    timestamp: str # 故事内绝对时间
    character_id: str # 关联 Canonical ID
    event_type: str # INJURY / HEAL / ITEM_GAIN / KNOWLEDGE
    description: str
    causal_pressure: float # 因果压强 0.1~1.0
```

## 四、 核心引擎流设计

### 4.1 上下文组装器
这是 Actor 代理的心脏，负责将三层数据打包送入 LLM。
1. **Token 预算分配**：设定总预算 8000 Tokens。
2. **人格注入**：加载 `prompts/actor.v1.md`。
3. **权威分层加载**：
   - `[CANON | IMMUTABLE]`：从 LlamaIndex 检索相关世界观规则（占比 20%）。
   - `[STATE MEMORY | MEDIUM]`：读取当前 POV 角色的 YAML 状态（占比 30%）。
   - `[SUBCONSCIOUS | LOW]`：从潜意识池检索相似灵感碎片（占比 10%，最易被裁剪）。
4. **近期正文**：加载上一段文本作为续写锚点（占比 40%）。
5. **熔断机制**：若组装后 Token 数超限，按权威层级从低到高截断。

### 4.2 Commit 审阅流
这是 Auditor 代理的核心，确保零信任落地。
1. **快照生成**：复制当前章节和角色的 YAML，存入 `.snapshots/`。
2. **提取请求**：将正文发送给 LLM，要求输出 JSON 格式的事件列表。
3. **Pydantic 校验**：若 JSON 格式不符 `EventLog` 模型，直接打回要求重试。
4. **Diff 展示**：使用 `Rich` 库在终端渲染绿/红色彩差异（如 `+ 左臂骨折`）。
5. **人工守门**：等待用户输入 `y` (写入), `n` (放弃), `e` (修改 JSON)。
6. **状态写入**：确认后，更新 Frontmatter 并插入 SQLite 记录。

## 六、 防御性边界条件 (防爆设计清单)
1. **并发写入冲突**：AI 流式输出时只写入内存缓冲区，按确认键后才调用 `python-frontmatter` 合并追加，绝不覆盖用户光标正在编辑的内容。
2. **API 雪崩**：所有 LLM 调用必须包裹 `tenacity` 重试机制（速率限制/超时重试）。
3. **状态污染**：严格执行 `loom rollback` 机制，任何一次 `commit` 前必须生成 Snapshot，出错可秒级恢复。
4. **幻觉降级**：上下文权威分级机制，确保 AI 绝不会把潜意识池的灵感当作世界观设定来执行。
